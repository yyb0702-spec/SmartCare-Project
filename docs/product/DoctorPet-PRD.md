## 1. 개요

반려동물 증상을 입력받아 AI가 방문 필요성과 필요 진료역량을 안내하고, 근처 동물병원을 찾아 예약·진료·후불 결제까지 이어주는 서비스다.

흐름은 이렇다: 증상 입력 → AI가 방문 필요성 + 필요 진료역량(requiredCapabilities)을 구조화하고 검색 Tool을 직접 호출 → 제휴/비제휴 2계층으로 노출된 병원 중에서 선택 → 예약 요청(REQUESTED) → 병원 승인(CONFIRMED) 또는 거절(REJECTED) → 체크인 → 진료중 → 진료완료 → 등록 빌링키로 후불 청구.

기술적으로 어려운 지점은 세 곳이다.

- **슬롯 동시 예약 충돌** — 같은 시간대에 여러 요청이 몰릴 때 하나만 성립해야 한다.
- **노쇼 처리** — 그레이스타임 경과 후 자동/수동 판정이 섞인다.
- **후불 결제 정합성** — 진료 후에야 금액이 확정되므로 검증·실패 분기·이중 청구 방지가 핵심이다.

MVP 전제는 다음과 같다.

- 예약은 병원 승인형이다. 병원 기능은 프론트 화면 없이 운영 API로만 제공한다.
- 축종(개/고양이)은 MVP부터 검색 필터(`supportedSpecies`)로 적용한다.
- 병원 데이터는 공공데이터 시드 적재(MVP 1회) + 제휴/비제휴 2계층 노출로 구성한다. 실시간 API 프록시는 쓰지 않는다.
- 노쇼는 판정·기록만 한다. 수수료나 예약 제한은 두지 않고, 예약 이력을 병원 승인 판단 자료로 넘긴다.

## 2. 문제와 목표

보호자는 증상이 나타났을 때 병원 방문이 필요한지, 어떤 진료역량을 갖춘 병원에 가야 하는지 판단하기 어렵다. 근처 예약 가능한 병원을 찾아 방문·진료·결제까지 이어가는 과정도 분절돼 있다. 진료비는 진료 내용에 따라 사후 확정되므로 사전 결제가 성립하지 않고, 예약 후 오지 않는 노쇼가 병원 운영을 방해한다.

풀어야 할 문제:

- 증상만으로는 방문 필요성도, 필요 진료역량도 판단하기 어렵다 → AI가 역량 단위로 구조화해서 검색까지 연결한다.
- 근처에서 조건에 맞는 예약 가능 병원을 찾기 번거롭다 → QueryDSL 조건 검색 + AI 추천이 같은 API를 공유한다.
- 동일 슬롯에 예약이 중복 확정되면 병원 운영이 불가능하다 → 동시성 제어로 막는다.
- 노쇼로 슬롯이 낭비되고 병원이 예약자 신뢰도를 판단할 근거가 없다 → 판정·기록하고 이력을 병원에 제공한다.
- 진료비가 사후 확정되어 사전 결제가 안 된다 → 빌링키 후불 청구로 해결한다.
- 결제 실패 시 대금 회수 경로가 불명확하다 → 원인별 분기 + 오프라인 정산으로 처리한다.
- 예약부터 결제까지 상태가 흩어지면 추적이 안 된다 → 상태를 일관되게 관리한다.

수의사법 제10조·제12조는 수의사가 아닌 자의 진료 행위와 처방을 금지한다. 반려동물 AI 모니터링은 규제샌드박스 실증특례로 일부 허용되지만 재진 모니터링에 한정되고 처방전 발급은 안 된다. 이 서비스의 AI는 "진단"·"처방"을 명시적으로 배제한다. `requiredCapabilities`는 병명이 아니라 검색 필터값이라 이 리스크 영역에 해당하지 않는다.

## 3. 사용자와 요구사항

**보호자**는 반려동물 증상이 있을 때 방문 여부와 병원을 찾는다. 필요한 건: 방문 필요성·필요 진료역량 안내(진단·처방 없음), 조건(지역·역량) 기반 검색과 예약, 예약 취소(2시간 전까지), 후불 자동 결제와 결과 통지. 예약은 요청 후 병원 승인을 거쳐야 확정되며, 승인 전에는 "예약 요청됨"으로만 보인다.

**병원 스태프**는 예약 요청을 검토·승인/거절하고, 체크인·진료 상태를 관리하고, 진료비를 청구한다. 결제가 최종 실패하면 오프라인으로 수납하고 시스템에 기록한다. 그레이스타임 전에도 노쇼를 수동 확정할 수 있고, 자동 판정을 정정할 수도 있다. 병원 기능은 프론트 화면 없이 운영 API로만 제공한다 — 요청 조회/승인/거절/체크인/진료 시작/완료/진료비 입력/오프라인 정산이 전부다.

**스케줄러**는 그레이스타임 경과 건과 병원 미응답 건을 자동으로 정리한다.

회원/인증: 가입은 이메일(또는 휴대폰)·비밀번호·닉네임만 받는다. 반려동물 정보는 가입 스키마에서 빼고 PetProfile로 분리해 선택 등록한다. Access Token은 30분~1시간, Refresh Token은 14일이며 서버 측에 저장·회전한다. 폐기된 Refresh Token이 재사용되면 해당 사용자 전체 세션을 로그아웃시킨다. 미인증은 401, 권한 없음은 403.

## 4. 핵심 가치와 차별점

증상을 병명이 아니라 "필요한 진료역량"으로 번역해서 방문 판단을 돕는다. AI 추천과 사용자 직접 검색이 같은 QueryDSL 검색 API를 공유하기 때문에, AI가 내놓은 결과를 그대로 받아도 되고 조건을 조정해 재검색해도 된다. 제휴 병원은 예약 가능, 비제휴 병원도 참고용으로 함께 보여줘서 "우리 동네엔 병원이 없다"는 공백을 없앤다.

거래 쪽에서는 동시성 제어로 슬롯 중복 확정을 막고, 병원이 예약자 이력을 보고 직접 승인/거절하는 실제 운영 구조를 갖는다. 진료비가 사후 확정되는 후불 구조를 빌링키로 처리하되, 실패 원인별로 분기하고 오프라인 정산으로 회수 경로를 남긴다. AI는 진단·처방을 배제하고 disclaimer·응급 라우팅으로 오정보 위험을 방어한다.

프로젝트 관점에서는 QueryDSL 동적 검색, 캐싱, 동시성, AI 안전, 결제 정합성이라는 백엔드 핵심 역량을 하나의 도메인에서 다룬다. 병원은 이력 기반으로 예약을 통제하고, 후불 결제로 현장 수납 부담을 던다.

## 5. 주요 이용 시나리오

### 5-1. 보호자 정상 흐름

```
증상 발생
  → 증상 입력 (프로필 없이 임시 정보로도 가능)
  → AI: 방문 필요성 + 필요 진료역량(requiredCapabilities) 구조화
  → AI: 검색 Tool 직접 호출 → QueryDSL로 제휴 병원 조회
  → AI: 실제 결과 근거로 자연어 안내 + 병원 목록
  → 병원 선택 (또는 검색 화면에서 조건 직접 조정 후 재검색)
  → 반려동물 프로필 + 빌링키 등록
  → 예약 요청(REQUESTED)
  → 병원이 예약자 이력 확인 후 승인 → 확정(CONFIRMED)
  → 병원 방문 → 체크인 → 진료중 → 진료완료
  → 병원이 최종 진료비 입력 → 등록 빌링키로 자동 결제
  → 결제완료 + 결과 알림
```

### 5-2. AI 진료역량 추천 (3단계)

```
"강아지가 어제부터 뒷다리를 절어요. 인천인데 갈 만한 병원 있어?"

1. 구조화 출력   증상 → { requiredCapabilities: ["엑스레이","정형외과 진료"], urgencyLevel: "MODERATE", ... }
2. Tool Calling  searchNearbyVets(requiredCapabilities=[...], region="인천") → QueryDSL 조회
3. 자연어 응답   "엑스레이·정형외과 진료가 가능한 병원이 인천 근처에 3곳 있어요. OO동물병원이 1.2km로 가장 가깝습니다."
```

AI 추천과 사용자 직접 검색은 같은 검색 API를 쓴다.

### 5-3. 응급 신호

증상 입력에 응급 키워드가 있으면 LLM 판단을 우회하고 고정 응답으로 "병원 즉시 방문"을 라우팅한다.

### 5-4. 예약 예외 상황

| 상황 | 처리 |
| --- | --- |
| 프로필 없이 예약 요청 | 요청 거부, 프로필 등록 화면으로 유도 |
| 빌링키 미등록 상태로 예약 요청 | 요청 거부, 카드 등록으로 유도 |
| 예약시각 4시간 이내 임박 슬롯 요청 | 요청 거부 (최소 리드타임 미충족) |
| 동일 슬롯 동시 예약 요청 | 1건만 성립, 나머지는 "다른 이용자가 방금 예약했습니다" |
| 병원이 예약 요청 거절 | REJECTED, 슬롯 반환 + 알림. 후불 구조라 환불 불필요 |
| 승인 데드라인까지 병원 미응답 | 타임아웃 자동 거절. 데드라인 = `min(요청+1시간, 예약시각−2시간)` |
| 비제휴 병원 | 참고용으로만 노출, 예약 버튼 없음 + "제휴 전 병원입니다" 배지 |
| 검색/예약 API 타임아웃 | LLM에 실패 컨텍스트 전달 → "지금은 실시간 예약 확인이 어려워요" |

> **[결정 필요 — 문서 충돌]** 위 검색 Tool 실패 처리는 SA §8-4·§9-5의 `fallback=true` 직접 검색 유도와 응답 생성 주체가 일치하지 않는다. LLM 재호출 여부와 서버 고정 문구 사용 여부는 팀 합의 후 PRD·정책·SA를 함께 수정하며, 결정 전에는 해당 실패 응답 방식을 구현하지 않는다.

### 5-5. 노쇼

예약시간 +10분까지는 그레이스타임으로 체크인을 받는다. 그 이후 미체크인이면 스케줄러(1분 주기)가 자동으로 NO_SHOW 처리하거나, 병원이 수동으로 확정한다(자동보다 우선). 병원이 "사실 도착했었다"고 정정하면 CHECKED_IN으로 되돌리고 이력에 남긴다. 노쇼는 기록만 되고, 이후 예약 요청 시 병원에 이력으로 제공된다 — 수수료나 제한은 없다.

### 5-6. 결제 실패 (원인별 분기)

```
진료 완료 → 빌링키 결제 승인 요청 → 실패
  재시도 유효(네트워크·타임아웃·일시 장애)
    → 단건 조회로 실제 처리 여부 확인 → 미처리면 제한 횟수 재시도 → 그래도 실패면 오프라인 전환
  재시도 무의미(카드 한도초과·정지·빌링키 만료)
    → 즉시 OFFLINE_REQUIRED 확정
  → 병원 현장 수납 후 오프라인 정산 기록
  → Payment = OFFLINE_PAID (예약은 진료완료 유지, 전체 결제완료는 예약+결제 조합으로 표현)
  ※ 오프라인 기록 후 자동 재시도 파이프라인 중단 (이중 청구 방지)
```

### 5-7. 병원 정상 흐름

예약 요청 목록 확인 → 예약자 이력 참고해 승인/거절 → 체크인 → 진료 시작·완료 → 진료비 입력해 자동 청구 → (실패 최종 확정 시) 오프라인 수납 후 정산 기록.

## 6. 예약과 결제 운영 정책

### 6-1. 병원 검색·데이터 소스

병원 정보는 동물병원 공공데이터(공공데이터포털·행안부 지방행정인허가데이터)를 쓴다. 확보되는 필드는 병원명·전화번호·주소·좌표·인허가일자·영업상태(영업/휴업/폐업)·휴폐업일자·관리번호·지자체코드 등이고, 영업시간과 진료가능종은 원본에 없다. 실시간 프록시가 아니라 자체 DB에 적재해 쓰고, 검색은 항상 자체 DB를 대상으로 한다. MVP는 특정 지자체(서울 등 1~2곳) 1회 시드 적재로 시작하고, 주기 갱신 배치는 확장에서 도입한다(SA §9-6).

매칭은 이름 문자열이 아니라 관리번호·지자체코드로 더미 제휴 데이터를 조인한다. 노출은 2계층이다 — 제휴 병원은 영업시간·진료가능종·슬롯을 자체 필드로 보강해 예약 가능, 비제휴 병원은 원본 정보만 참고용으로 노출하고 예약 버튼 없이 배지만 붙는다. 제휴 병원이 충분히 늘면 공공데이터 의존도를 낮추는 방향을 재검토한다.

### 6-2. 검색 — QueryDSL + AI Tool Calling

검색 엔진은 자체 DB 대상 QueryDSL 동적 쿼리다(다중 조건 조합·페이징·정렬). 조건 예: 지역·거리·진료역량·검사·처치·수술/입원/야간/응급 여부·장비·운영시간. AI는 이 검색 API를 Tool로 직접 호출하며 사용자 직접 검색과 API를 공유한다. 축종 필터(`supportedSpecies`)는 MVP부터 실제 필터로 적용한다.

### 6-3. 예약 상태 전이

```
요청됨(REQUESTED)
   → 승인 → 확정(CONFIRMED) → 체크인(CHECKED_IN) → 진료중(IN_TREATMENT) → 진료완료(TREATMENT_COMPLETED)
   → 거절(REJECTED)   [병원 거절, 또는 승인 데드라인까지 미승인]
   → 취소(CANCELED)   [사용자 취소, 예약시각 2시간 전까지]
CONFIRMED → 취소(CANCELED)   [예약시각 2시간 전까지]
CONFIRMED → 노쇼(NO_SHOW)    [체크인 없이 그레이스타임 경과]
```

예약은 프로필+빌링키 등록 완료 상태에서, 예약시각 최소 4시간 전까지만 요청할 수 있다. 승인 전에는 "예약 요청됨"으로만 보인다. 승인 데드라인은 `min(요청시각+1시간, 예약시각−2시간)` — 고정 시간만 쓰면 임박 슬롯이 즉시 판단을 강요받고, 슬롯 기준만 쓰면 먼 미래 예약은 병원이 며칠씩 응답을 미룬다. 이른 쪽을 택해 둘 다 막는다. 취소도 예약시각 2시간 전까지 가능하고, REQUESTED·CONFIRMED 두 상태 모두 해당한다. 거절 사유는 정형화한다: 직원 부족 / 슬롯 등록 오류 / 진료 불가 / 기타.

### 6-4. 동시성 (필수 과제)

동일 슬롯 동시 예약 요청은 1건만 성립해야 한다. 조건부 UPDATE / 낙관적 락 / Redis 분산 락 중 택1이며, 단일 슬롯 경합엔 조건부 UPDATE가 가장 단순하다. 실제 비교·선정은 SA에서 다룬다 → **SA §9-3에서 낙관적 락으로 확정했다.** 실패 시 "다른 이용자가 방금 예약했습니다"로 안내하고, 다중 스레드 시나리오로 중복 성립 0건을 검증한다.

### 6-5. 취소

예약시각 2시간 전까지는 REQUESTED·CONFIRMED 모두 자유롭게 취소할 수 있다. 그 이후엔 취소가 거부되고 미도착 시 노쇼 대상이 된다. 병원 사정 취소는 REJECTED로 처리하며 사용자 페널티는 없다.

### 6-6. 노쇼 — 수수료·누적제한 없음, 이력 제공형

그레이스타임은 예약시간 +10분이다. 스케줄러(1분 주기)가 경과 시 NO_SHOW 처리하되, 병원 수동 판정이 항상 우선이다. 정정 시 CHECKED_IN으로 재전이하고 이력을 남긴다. 수수료도, 누적 제한도 두지 않는다. 대신 예약 요청 시 병원에 예약 이력을 제공해 승인 판단에 쓰게 한다.

```json
{
  "reservationHistory": {
    "totalReservationCount": 15,
    "completedCount": 11,
    "cancelCount": 1,
    "noShowCount": 3
  }
}
```

수수료를 도입하면 금액 산정·사전 동의·오판정 정정·환불·분쟁 처리 같은 부수 정책이 줄줄이 따라온다. 그 복잡도를 병원 승인 판단으로 대체한다. 이력 집계는 특정 병원이 아니라 전 병원 통합이다 — 병원이 "이 사용자가 다른 병원 포함해서 얼마나 노쇼하는가"를 판단할 수 있어야 하기 때문이다. 자동 차단은 하지 않는다.

### 6-7. 결제 (PortOne 빌링키 — 후불형)

예약 요청/확정 시점엔 빌링키만 등록한다(승인·capture 없음). 진료 완료 후 병원이 최종 진료비를 입력하면 등록된 빌링키로 결제를 승인 요청한다. 청구 금액은 0 초과 & 서비스 절대 상한 이하만 허용한다(병원 오입력 방어. 상한값은 SA 참조).

결제 상태는 `PENDING`(청구 시작) → `PAID`(승인 성공) / `OFFLINE_REQUIRED`(자동 결제 불가) → `OFFLINE_PAID`(현장 수납 완료)로 흐른다. 승인 요청 후 응답이 유실되면 별도 상태 없이 `PENDING`에 머문다 — 이미 승인됐을 수 있어 무조건 재시도는 이중 청구 위험이다. 정산 스케줄러가 일정 시간 후 PortOne 단건 조회로 `PAID` 또는 `OFFLINE_REQUIRED`를 확정한다.

실패 원인은 둘로 나뉜다. 네트워크 오류·타임아웃·일시 장애 같은 재시도 유효 건은 단건 조회 후 제한 횟수만큼 재시도하고, 그래도 실패면 `OFFLINE_REQUIRED`로 넘긴다. 카드 한도초과·정지·빌링키 만료처럼 재시도가 무의미한 건은 즉시 `OFFLINE_REQUIRED`로 확정한다. 최종 실패가 예약·진료 기록을 되돌리지 않는다 — 진료완료 상태는 유지되고 병원 화면에 미수금으로 뜬다. 회수는 서비스가 자동 보장하지 않고 병원이 오프라인으로 처리하는 게 원칙이다.

오프라인 정산은 `PATCH /api/hospital/payments/{paymentId}/offline-settle`로 처리한다. 전제조건은 `Payment.status == OFFLINE_REQUIRED`이며, 처리 후 `OFFLINE_PAID`로 바뀌고 `paymentChannel = OFFLINE`(자동결제는 `BILLING_KEY`)로 구분 기록된다. 이 이후엔 자동 재시도 파이프라인을 반드시 중단시켜야 이중 청구가 안 난다.

환불은 MVP 범위 밖이다. 후불 구조라 취소·거절·노쇼 시점엔 애초에 청구가 없어 환불이 필요 없고, 유일한 케이스인 오청구는 병원 수기·운영자 콘솔로 처리한다. 전액/부분 환불·정정 재청구는 확장에서 도입한다.

클라이언트가 보낸 결제 결과는 신뢰하지 않고 서버가 PortOne 단건 조회로 금액·상태를 검증한다. 빌링키는 암호화 저장하고 카드 원본은 저장하지 않는다.

## 7. AI 진료역량 추천·안전 정책

AI는 병명이나 진료과를 판단하지 않는다. 증상을 "이 병원이 갖추면 좋은 진료 역량" 단위로 구조화하고, 그 값으로 검색 Tool을 호출한다. 최종 선택과 조건 조정은 사용자가 검색 화면에서 직접 한다.

```json
{
  "possibleFocusAreas": ["소화기", "피부"],
  "requiredCapabilities": ["엑스레이", "정형외과 진료"],
  "urgencyLevel": "MODERATE",
  "preVisitCheckpoints": ["최근 식이 변화 여부", "구토·설사 지속 시간"],
  "recommendVetVisit": true
}
```

의심 질환명이나 처치 지침은 스키마에 없다. `requiredCapabilities`는 진단이 아니라 검색 필터값이다. `disclaimer`는 이 스키마에 두지 않는다 — LLM이 누락할 수 있으므로 서버가 응답을 조립하는 마지막 단계에서 고정 문구를 항상 주입한다.

연동은 3단계다. 증상을 스키마로 구조화하고(구조화 출력), `requiredCapabilities`와 위치를 인자로 검색 API를 Tool로 직접 호출하고(Tool Calling), 조회된 실제 DB 결과만 근거로 자연어 응답을 만든다(그라운딩·없는 데이터 생성 차단).

환각·오정보 방어: disclaimer는 서버가 항상 주입한다. 응급 키워드가 감지되면 LLM 판단을 우회하고 "병원 즉시 방문" 고정 응답으로 라우팅한다. 화이트리스트 카테고리 밖은 "정확한 답변이 어렵습니다"로 답한다. `urgencyLevel: HIGH`는 처치성 문구 대신 병원 연결로 강제 분기한다. 질환명 추정·약물·처치 지시는 생성 자체를 차단한다. AI 장애 시엔 사용자가 직접 검색 조건을 고르는 QueryDSL 검색으로 대체한다.

프롬프트는 최소 2개 축으로 분기한다 — 종(개/고양이)별, 증상 카테고리(소화기/피부/호흡기 등)별. temperature는 0.2~0.3으로 낮게, max_tokens는 500 정도로 스키마 길이에 맞춘다.

수의사법 제10조·제12조는 무면허 진료·처방을 금지하고, 규제샌드박스 실증특례는 재진 모니터링에 한정된다. `requiredCapabilities` 추천은 진단이 아니라 검색 필터이므로 이 리스크 영역 밖이다.

## 11. 문서 범위 밖 (SA 문서에서 정의)

ERD, API 명세, 상태 머신 상세, QueryDSL 검색 설계, 공공데이터 배치·2계층 매핑, 동시성 제어 비교·선정, 결제 실패 분류·재시도·오프라인 정산 파이프라인, 캐싱 전략, AI 프롬프트 외부화·장애 격리, 핵심 시퀀스 다이어그램.
