## 1. 회원·인증
- 회원 유형은 보호자(`GUARDIAN`)와 병원 스태프(`HOSPITAL_STAFF`)로 구분한다.
- 반려동물은 회원가입 정보에서 제외하고 가입 후 선택 등록한다. 단, 예약 요청 전에는 프로필 등록이 필수다.
- Access Token 수명은 30분\~1시간, Refresh Token 수명은 14일이다. Refresh Token은 서버에 저장하고 회전하며, 폐기된 토큰 재사용 시 해당 사용자의 전체 세션을 로그아웃한다.
- 탈퇴 시 이메일을 `withdrawn_{memberId}@deleted.doctorpet` 형식으로 익명화하고 Soft Delete한다. 활성 회원만 로그인·이메일 중복검사 대상으로 삼아 원 이메일 재가입을 허용한다.
## 2. 병원·검색
- 병원 기본정보는 공공데이터를 자체 DB에 적재해 사용한다.
- 제휴 병원은 `hospitals.partnership_status=PARTNER`로 판별한다. 공공데이터와 제휴 데이터는 지자체 코드와 관리번호(`local_gov_code + mgmt_no`) 복합 키로 매핑하며, 두 컬럼에 복합 UNIQUE를 적용한다.
- 제휴 병원은 상세정보·진료역량·예약을 제공하고, 비제휴 병원은 참고 정보만 제공한다.
- 검색 조건은 지역, 거리, 동물 종, 진료 분야, 검사, 처치, 수술·입원·야간·응급 여부, 장비, 운영 시간이다.
- 검색은 자체 DB를 대상으로 QueryDSL 동적 쿼리로 처리하며, 전달된 조건만 조합하고 페이징·정렬을 지원한다.
- 상세 진료역량 조건이 있으면 요청 조건을 모두 보유한 제휴 병원만 검색한다.
- 폐업 병원은 기본 검색에서 제외하고 휴업 병원은 상태를 표시한다.
- MVP는 특정 지자체 데이터를 1회 시드 적재하며, 주기 갱신 배치는 확장 범위다.
## 3. 예약·진료
### 예약 요청
- 반려동물 프로필과 빌링키가 등록되어 있어야 한다.
- 예약 시각 최소 4시간 전까지만 요청할 수 있다.
- 예약은 `REQUESTED`로 생성하며 병원이 승인하면 `CONFIRMED`, 거절하면 `REJECTED`가 된다.
- 병원 응답 기한은 `min(요청시각+1시간, 예약시각-2시간)`이다. 기한 경과 시 자동 거절하고 슬롯을 반환한다.
- 거절 사유는 직원 부족, 슬롯 등록 오류, 진료 불가, 기타의 4종으로 관리한다.
- 예약 취소는 `REQUESTED` 또는 `CONFIRMED` 상태에서 예약 시각 2시간 전까지만 허용하며 슬롯을 반환한다.
### 진료
```plain text
CONFIRMED → CHECKED_IN → IN_TREATMENT → TREATMENT_COMPLETED
```
- 병원 스태프는 소속 병원의 예약만 처리한다.
- 진료 완료와 진료비 청구는 별도 요청으로 처리한다.
## 4. 노쇼
- 예약 시각 +10분까지는 지각으로 간주하여 체크인을 허용한다.
- 10분을 초과한 미체크인 예약은 자동으로 `NO_SHOW` 처리한다.
- 병원의 수동 판정이 자동 판정보다 우선한다.
- 자동 노쇼 후 병원이 오판정을 정정하면 `CHECKED_IN`으로 복구하고 이력을 추가한다.
- 노쇼 시 슬롯은 반환하지 않고 `RESERVED` 상태로 유지한다. 예약 시각이 지나 재판매 가치가 없기 때문이다.
- 병원에는 해당 사용자의 전 병원 통합 예약 이력을 제공한다.
- 이력은 전체 예약, 진료 완료, 취소, 노쇼 횟수로 구성하며 정정된 노쇼는 노쇼 횟수에서 제외한다.
- 노쇼 수수료와 누적 노쇼에 따른 서비스 전체 자동 예약 제한은 적용하지 않는다.
## 5. 결제
- 보호자는 예약 요청 전에 빌링키를 등록한다. 등록 시 금액 승인은 발생하지 않는다.
- 진료 완료 후 병원이 최종 진료비를 입력하면 등록된 빌링키로 결제한다.
- 진료비는 0원 초과, 300만원 이하만 허용한다. 상한은 코드 상수가 아닌 설정값으로 관리한다.
- 결제 전 서버가 `PENDING` 기록과 고유한 `merchant_payment_id`를 먼저 생성한다.
- 타임아웃으로 결과가 불확실하면 재청구하지 않고 단건 조회로 결과를 확인한다.
- 네트워크 오류와 일시 장애는 단건 조회 후 미처리 건에 한해 최대 3회 지수 백오프로 재시도한다.
- 한도 초과, 카드 정지, 빌링키 만료·삭제는 재시도하지 않고 즉시 `OFFLINE_REQUIRED`로 전환한다.
- 오프라인 정산은 `OFFLINE_REQUIRED` 상태에서만 허용한다.
- 정산 후 `OFFLINE_PAID`와 처리 시각·처리자를 기록하고 자동 재시도 파이프라인을 중단하여 이중 청구를 방지한다.
- 환불과 정정 후 재청구는 MVP에서 제외한다.
## 6. AI 진료역량 추천
- 비로그인 사용자도 임시 증상 정보로 AI 상담을 이용할 수 있다.
- AI는 진단·처방하지 않고 증상을 필요한 진료역량으로 구조화한 뒤 실제 병원 검색 Tool을 호출한다.
- 구조화 출력은 다음 5개 필드를 사용한다.
```json
{
  "possibleFocusAreas": ["소화기", "피부"],
  "requiredCapabilities": ["엑스레이", "정형외과 진료"],
  "urgencyLevel": "MODERATE",
  "preVisitCheckpoints": ["최근 식이 변화 여부", "구토·설사 지속 시간"],
  "recommendVetVisit": true
}
```
- 응급 키워드는 LLM 판단을 우회하고 고정 응답으로 즉시 병원 방문을 안내한다.
- AI가 허용된 화이트리스트 밖의 카테고리를 생성하면 `"정확한 답변이 어렵습니다"`로 응답한다.
- **\[결정 필요 — 정본 충돌\]** 검색 Tool 실패 시 PRD·정책은 실패 컨텍스트를 LLM에 전달해 안내 문구를 생성하도록 하고, SA는 `fallback=true`로 직접 검색을 유도한다. LLM 재호출 여부와 서버 고정 문구 사용 여부는 추후 팀 합의 후 정본을 함께 수정하며, 결정 전에는 구현하지 않는다.
- 검색 결과에 없는 병원·질환명·처치 지시를 생성하지 않는다.
- 면책 문구는 LLM 출력이 아니라 서버가 항상 주입한다.
- 증상 원문은 전화번호·이메일·주민번호 등 패턴을 마스킹해 30일간 보존하고, 30일이 지난 원문은 배치로 삭제한다. 배치 주기와 구현 방식은 정본에 확정되어 있지 않다.
