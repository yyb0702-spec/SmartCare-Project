## 0. 서비스 개요

사용자는 검색 조건을 직접 선택하거나, AI에게 증상을 자연어로 입력해 필요 진료역량을 추천받을 수 있다. 두 경로 모두 동일한 QueryDSL 기반 동적 검색 API를 거쳐 제휴 동물병원을 조회하고, 병원이 예약 요청을 승인/거절하는 구조로 예약이 확정된다. 이후 체크인·진료·후불 결제(BillingKey)까지 이어진다.

거래의 핵심 문제: 예약 선점 충돌, 병원 승인 흐름, 노쇼 이력 기반 판단, 진료 후 후불 결제 정합성

### 1.2 JWT 정책

| 구분 | 내용 |
| --- | --- |
| Access Token | 짧은 만료 (30분~1시간) |
| Refresh Token | 긴 만료 (14일), 서버 측 저장 및 회전(rotate) |
| 재사용 감지 | 폐기된 Refresh Token 재사용 시 해당 사용자 전체 세션 로그아웃 |

- 인증 실패 테스트 케이스: 만료 토큰 / 위조 토큰 / 재사용된 Refresh Token

> 참고: 팀원 제안서 13장에서 Refresh Token 회전을 확장 기능(MVP 이후)으로 미루자는 의견이 있었다. 다만 이 항목은 이번 AI/공공데이터 쟁점과 별개로 별도 논의된 적이 없어, 이번 개정에서는 기존 결정(MVP 포함)을 유지했다. 필요하면 팀 논의 후 별도로 결정.

## 3. 병원 검색·예약 정책

### 3.1 데이터 소스 — *(v6 결정 유지: 공공데이터·OpenAPI 계속 사용)*

- 병원 정보는 동물병원 전용 공공데이터(공공데이터포털 표준데이터셋)를 배치 수집해 자체 DB에 적재하고, 사업장 고유 식별자로 더미 제휴 데이터와 매칭한다.
- 노출은 2계층: 제휴 병원(예약 가능, 상세정보 보강) / 비제휴 병원(참고용, 예약 버튼 없음).
- 이 결정은 팀원 제안서 5장(OpenAPI 제외)을 반영하지 않은 부분이며, 별도 반박 문서(`정책변경제안서_검토의견.md`)에 근거를 정리해두었다.

### 3.2 병원 데이터 상세화 — *신규 반영 (제안서 4장)*

병원 엔티티에 아래 필드를 추가한다. 필드 증가 자체는 상태 전이·동시성·결제처럼 구현 난이도를 높이는 요소가 아니라, 검색 정확도를 높이는 요소로 간주한다.

| 분류 | 필드 |
| --- | --- |
| 기본 정보 | 병원명, 주소, 전화번호, 위치 좌표, 운영 시간 |
| 진료 정보 | 진료 가능 동물 종, 진료 가능 분야, 가능한 검사, 가능한 처치 |
| 시설 정보 | 수술 가능 여부, 입원 가능 여부, 야간 진료 여부, 응급 진료 여부, 보유 의료 장비 |

> 진료역량·시설·축종 상세 필드는 **제휴 병원만** 보강한다. 따라서 역량/축종/시설 조건으로 검색하면 비제휴 병원은 결과에서 빠지고, 비제휴는 지역·거리 등 원본 필드 기반으로만 참고 노출된다.

### 3.3 QueryDSL 기반 동적 검색 — *신규 반영 (제안서 3장)*

- 검색 조건: 지역, 거리, 진료 가능 동물 종, 진료 가능 분야, 가능한 검사, 가능한 처치, 수술 가능 여부, 입원 가능 여부, 야간 진료 여부, 응급 진료 여부, 보유 장비, 운영 시간
- 조건 조합은 QueryDSL 동적 쿼리로 생성하며, 페이징·정렬을 지원한다.
- **종(species) 필터링은 v6에서 확장 항목으로 분류했으나, 이번 개정에서 MVP 검색 조건으로 편입한다.** (`supportedSpecies` 필드는 확장 대비가 아니라 지금부터 실제 필터 조건으로 사용)
- AI 경로(4장)도 동일한 검색 API를 Tool Calling으로 호출하므로, 검색 조건 확장은 AI 추천 정확도에도 그대로 반영된다.

### 3.4 예약 동시성 제어

- 동일 슬롯 동시 예약 시도 → **조건부 UPDATE(`WHERE status='OPEN'`) / 낙관적 락(버전 컬럼) / Redis 분산 락 중 택1** (단일 슬롯 경합엔 조건부 UPDATE가 가장 단순·확실하며, 비교·선정 근거는 SA에서 문서화)
- 실패 시 안내: "다른 이용자가 방금 예약했습니다"

### 3.5 예약 상태 전이 — *변경: 병원 승인형 예약을 MVP로 편입 (제안서 6, 8, 9장)*

즉시 확정 방식을 폐기하고, 병원이 승인/거절하는 구조를 MVP 기본 흐름으로 확정한다.

```
REQUESTED(예약 요청) → CONFIRMED(병원 승인) → CHECKED_IN(체크인) → IN_TREATMENT(진료중) → TREATMENT_COMPLETED(진료완료, 종착)
                     ↘ REJECTED(병원 거절 또는 타임아웃)
CONFIRMED →            ↘ NO_SHOW(체크인 없이 그레이스타임 경과)
              ↘ CANCELED(사용자 취소, 3.7)
```

- **REQUESTED 생성 조건**: 빌링키 등록 완료 + 반려동물 프로필 등록 완료 + **예약시각 최소 4시간 전까지만 요청 가능**(임박 슬롯은 요청 거부). 예약 요청 자체의 전제조건이며, 승인 여부와 별개다.
- **병원 승인 판단 근거**: 담당 수의사 일정, 병원 내부 사정, 해당 동물 진료 가능 여부, 장비·입원공간 여력, 응급환자 발생, **사용자의 과거 예약 이력(6.2)** 등 시스템이 알 수 없는 정보를 병원이 직접 판단한다.
- **체크인/진료 처리**: 별도 기능이 아니라 병원 운영 API(3.8)를 통한 상태 변경으로 처리한다.
- **결제완료 표시**: 예약 상태가 아니라 **Payment 상태**로 표현한다 — 자동 결제 성공(`PAID`) 또는 오프라인 정산 기록(`OFFLINE_PAID`, 5.4). 화면의 "결제완료"는 진료완료(예약)+결제 상태의 **조합**으로 계산한다(SA §5-4).

### 3.6 병원 승인 타임아웃 정책

- **전제**: 최소 리드타임 4시간(§3.5)이 있어 요청 시점은 항상 예약시각 4시간 이전이다. 이 위에서 데드라인을 정한다.
- 병원 응답 데드라인은 **"요청 후 고정 유예시간"과 "예약 슬롯 시각 기준 사전 시간" 중 더 이른 시점**으로 정한다.
    - 예: `min(요청시각 + 1시간, 예약슬롯시각 − 2시간)`
    - 근거: 슬롯 기준만 쓰면 먼 미래 예약은 병원이 응답을 며칠씩 미뤄도 되어 사용자가 오래 대기하게 되고, 고정 시간만 쓰면 임박한 당일 예약은 병원이 준비할 시간 없이 즉시 판단을 강요받는다. 두 기준 중 이른 쪽을 택해 두 문제를 동시에 방지한다.
- 데드라인 경과 시 자동 REJECTED 처리한다 (응답없음=자동승인보다 안전).
- **거절 사유 정형화**: 직원 부족 / 슬롯 등록 오류 / 진료 불가(동물 특성상) / 기타

### 3.7 취소 정책

- 예약시간 2시간 전까지 자유 취소 가능 (REQUESTED/CONFIRMED 상태 모두 해당). 2시간 이내로는 취소가 거부되며, 이후 미도착 시 노쇼 판정 대상이 된다.
- 병원 사정으로 인한 취소(REJECTED 포함)는 사용자 페널티 없음

### 3.8 병원 운영 API — *신규 반영 (제안서 7장)*

병원 전용 프론트 화면은 MVP 필수가 아니다. 병원 역할은 API로만 구현하고 Swagger/Postman으로 검증한다.

| API | 설명 |
| --- | --- |
| `GET /api/hospital/reservations` | 예약 요청 조회 (사용자 예약 이력 포함, 6.2) |
| `PATCH /api/hospital/reservations/{reservationId}/approve` | 예약 승인 |
| `PATCH /api/hospital/reservations/{reservationId}/reject` | 예약 거절 (사유 포함) |
| `PATCH /api/hospital/reservations/{reservationId}/check-in` | 체크인 처리 |
| `PATCH /api/hospital/reservations/{reservationId}/start` | 진료 시작 처리 |
| `PATCH /api/hospital/reservations/{reservationId}/complete` | 진료 완료 처리 + 진료비 입력 |
| `PATCH /api/hospital/payments/{paymentId}/offline-settle` | 오프라인 결제 완료 처리 |

각 API 공통 검증 항목: 로그인 사용자가 병원 스태프인지, 해당 예약이 소속 병원 건인지, 현재 상태에서 전이 가능한 상태인지, 중복 요청인지.

> API 경로는 SA 명세를 정본으로 하며 `/api/hospital/**`(단수)로 통일한다.

### 3.9 Tool 실패 처리

- 검색/예약 API 타임아웃 등 실패 시 → 실패 컨텍스트를 LLM에 전달 → 자연어로 "지금은 실시간 확인이 어려워요, 잠시 후 다시 시도해주세요" 안내

> **[결정 필요 — 문서 충돌]** 위 방식은 SA §8-4·§9-5의 `fallback=true` 직접 검색 유도와 응답 생성 주체가 일치하지 않는다. LLM 재호출 여부와 서버 고정 문구 사용 여부는 팀 합의 후 PRD·정책·SA를 함께 수정하며, 결정 전에는 해당 실패 응답 방식을 구현하지 않는다.

## 4. AI 증상검색·방문 안내 정책

### 4.1 역할 정의

AI는 병명이나 진료과를 직접 판단하지 않는다. 증상을 "필요 진료역량" 단위로 구조화하고, 그 값으로 3.3의 검색 API를 Tool Calling으로 호출한다. 최종 병원 선택은 사용자가 직접 한다.

### 4.2 입력/출력 스키마

```json
{
  "possibleFocusAreas": ["소화기", "피부"],
  "requiredCapabilities": ["엑스레이", "정형외과 진료"],
  "urgencyLevel": "MODERATE",
  "preVisitCheckpoints": ["최근 식이 변화 여부", "구토·설사 지속 시간"],
  "recommendVetVisit": true
}
```

> **`disclaimer`는 LLM 출력 스키마에 두지 않는다.** LLM이 누락할 수 있으므로, 서버가 응답을 조립하는 마지막 단계에서 고정 문구("본 정보는 참고용이며 실제 진단을 대체하지 않습니다")를 항상 주입한다.

### 4.3 AI-백엔드 연동 흐름 (Tool Calling)

```
1단계 — 구조화 출력: 증상 → 4.2 스키마 (진단 아님)
2단계 — Tool Calling: requiredCapabilities, 위치 등을 인자로 3.3 검색 API 호출
3단계 — 자연어 응답: 실제 검색 결과 기반 안내
```

### 4.4 환각 방어 (필수)

- disclaimer **서버 주입**, 응급 키워드 감지 시 고정 응답 라우팅, 화이트리스트 기반 카테고리 제한, `urgencyLevel: HIGH` 강제 분기, 질환명·처치 지시 생성 차단

### 4.5 프롬프트 분기 · 모델 파라미터

- 종별/증상 카테고리별 분기 (최소 2개), `temperature` 0.2~0.3, `max_tokens` 제한

### 4.6 법적 리스크 노트

- 수의사법 제10조·12조 — 서비스 내 AI는 "진단"·"처방" 표현과 기능을 배제한다. `requiredCapabilities` 추천은 검색 필터이므로 이 리스크 영역에 해당하지 않는다.

## 5. 결제 정책 (PortOne 빌링키 — 후불 결제형)

### 5.1 결제 시점

| 시점 | 처리 |
| --- | --- |
| **예약 요청(REQUESTED) 전** | 빌링키 등록 (카드 등록, 승인/capture 없음) — **§3.5 REQUESTED 생성 전제조건**, 미등록 시 예약 요청 자체가 거부됨 |
| 진료 완료 시 | 병원이 최종 진료비 입력 → 등록된 빌링키로 결제 승인 요청 |

> 빌링키는 **예약을 요청하는 시점에 이미 등록되어 있어야** 한다(프로필과 함께 REQUESTED 전제조건). 승인(CONFIRMED)은 병원의 판단일 뿐 결제수단 등록 시점과 무관하다.

### 5.2 결제 실패 원인별 처리 — *변경 (제안서 11장)*

무조건 3회 재시도 방식을 폐기하고, 실패 원인에 따라 분기한다.

**결과 미확정(타임아웃) — 재시도 금지**
```
결제 요청 → 응답 유실 → 별도 상태 없이 PENDING 유지(무조건 재시도 시 이중 청구 위험)
        → 정산 스케줄러가 단건 조회로 PAID 또는 OFFLINE_REQUIRED 확정
```

**재시도가 의미 있는 경우** (네트워크 오류, 일시적 외부 결제 장애)
```
단건 조회로 실제 처리 여부 확인 → 미처리 시 제한된 횟수 재시도(지수 백오프) → 소진 시 OFFLINE_REQUIRED 전환
```

**재시도가 의미 없는 경우** (카드 한도 초과, 카드 정지, 빌링키 만료)
```
결제 실패 → 즉시 OFFLINE_REQUIRED 전환 (재시도 생략)
```

### 5.3 최종 상태 처리

- 진료 자체는 이미 완료된 상태이므로, 결제 최종 실패가 예약·진료 기록을 되돌리지 않는다.
- Payment.status = OFFLINE_REQUIRED로 남고, Reservation은 TREATMENT_COMPLETED에 머무르며 병원 화면에 미수금으로 노출된다.
- 회수는 병원이 오프라인으로 처리하는 것을 원칙으로 한다 (서비스가 자동 보장하지 않음).

### 5.4 오프라인 정산 처리 — *필드 확장 (제안서 11-3)*

**API**: `PATCH /api/hospital/payments/{paymentId}/offline-settle`

| 항목 | 내용 |
| --- | --- |
| 인가 | 로그인한 병원 스태프 == 해당 예약의 소속 병원 |
| 전제조건 | Payment.status == OFFLINE_REQUIRED |
| 처리 | Payment.status: OFFLINE_REQUIRED → OFFLINE_PAID |
| 저장 정보 | 결제 방식, 처리 병원, 처리 시간, 처리 담당자 |
| 이력 | append-only로 "오프라인 수납 처리" 이벤트 기록, 이후 자동 재시도 파이프라인 중단 |
| 연동 효과 | 예약은 TREATMENT_COMPLETED 유지 — "결제완료" 표시는 예약+결제 상태 **조합**으로 계산(SA §5-4) |

### 5.5 환불 — MVP 제외

- **환불은 MVP 범위에서 제외한다** (SA v1.4·PRD v3.4 정합).
- 후불 구조라 취소·거절·노쇼 시점에는 청구 자체가 없어 환불이 불필요하고, 유일한 케이스(진료 후 오청구)는 **병원 수기·운영자 콘솔**로 처리한다.
- 전액/부분 환불·정정 재청구는 **확장(7장)** 에서 결제 이력 구조와 함께 도입한다.

### 5.6 금액 검증·보안

- 진료비 `amount`는 **0 초과 & 서비스 절대 상한 이하**만 허용한다(병원 오입력 방어).
- 클라이언트 결제 결과를 신뢰하지 않고, 서버가 PortOne 단건 조회로 금액·상태를 검증한다.
- 빌링키 암호화 저장, 카드 원본 정보는 저장하지 않음.

## 6. 노쇼 정책 — *대폭 변경 (제안서 10장)*

노쇼 판정과 기록은 유지하되, **노쇼 수수료와 서비스 전체 자동 예약 제한은 제거**한다. 대신 노쇼 이력을 병원의 승인 판단 자료로 제공한다.

### 6.1 판정 방식과 기록

- 그레이스 타임: 예약시간 +10분까지 "지각"으로 간주, 체크인 가능. 10분 초과 시 자동 NO_SHOW 전환.
- 병원의 수동 판정이 항상 시스템의 자동 판정보다 우선한다 (그레이스타임 전 병원의 "노쇼 확정" 즉시 반영, 자동판정 후 정정 시 CHECKED_IN 재전이 허용 + 이력 append).
- NO_SHOW 시 슬롯은 예약시각이 지나 재판매 가치가 없으므로 반환하지 않고 소비 상태로 유지한다.
- **제거된 기능**: 노쇼 수수료 청구, 노쇼 3회 누적 시 서비스 전체 예약 자동 차단 — 이 두 가지는 수수료 기준·산정·동의·오판정 환불·분쟁처리 등 부가 정책이 계속 늘어나는 구조라 제거한다.

### 6.2 노쇼 이력 기반 병원 승인 지원 — *신규*

병원이 예약 요청(3.8 `GET /api/hospital/reservations`)을 조회할 때, 해당 사용자의 예약 이력을 함께 제공한다.

```json
{
  "reservationHistory": {
    "totalReservationCount": 15,
    "completedCount": 11,
    "cancelCount": 1,
    "noShowCount": 3
  }
}
```

- **이력 집계 범위는 전 병원 통합이다.** 특정 병원 기준이 아니라 해당 사용자의 서비스 전체 예약·완료·취소·노쇼 횟수를 제공한다 — 병원이 "이 사용자가 (다른 병원 포함) 얼마나 노쇼하는가"를 승인 판단에 쓰기 위함이다.
- 병원은 이 정보를 참고해 승인/거절(3.5)을 자체 판단한다. 시스템이 자동으로 차단하지 않는다.
- 효과: 병원이 상황에 맞게 판단 가능, 자동 제재로 인한 사용자 불만 감소, 결제/정책 복잡도 감소.
- 마이페이지에는 본인 이력을 그대로 노출해 투명성을 유지한다 (v6의 "제한까지 M회 남음" 문구는 제거, 단순 이력 표시로 대체).

## 7. 지금(MVP) vs 나중(확장) 범위 정리

| 항목 | 지금 (MVP) | 나중 (확장) |
| --- | --- | --- |
| 병원 데이터 | 공공데이터 배치 수집 + 상세 필드(진료역량/시설정보) 보강, 2계층 노출 | 실제 제휴 병원 API 연동 |
| 검색 | QueryDSL 다중 조건(종/분야/검사/시설/장비/운영시간) + 페이징/정렬 | 검색 성능 최적화(캐싱, 검색 인덱스) |
| AI 역할 | 필요 진료역량 추천 + 검색 Tool 호출 (진단·처방 배제) | 규제 상황에 따라 재검토 |
| 예약 확정 | 병원 승인형 (REQUESTED→CONFIRMED/REJECTED), 하이브리드 타임아웃 | — |
| 노쇼 | 판정 + 기록 + 이력 제공(병원 승인 참고자료) | — (자동 제한은 이번 개정에서 폐기) |
| 결제 실패 | 원인별 분기(재시도 vs 즉시 오프라인) + 오프라인 정산 | 자동 재청구 고도화, 부분 환불 |
| 환불 | (MVP 제외) | 오청구 전액/부분 환불·정정 재청구 |
| 실시간 알림 | notifications 저장 + 폴링 조회 | SSE/WebSocket push (도전 — 실시간 메시징) |
| 병원 운영 | API만 구현 (Swagger/Postman 검증), 프론트 없음 | 병원 전용 프론트 화면 |
| 기타 확장 | — | Refresh Token 회전 재검토(1.2 참고), 대기열, 리뷰, 병원 통계, 관리자 기능 |
