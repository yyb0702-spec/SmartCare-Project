# 구현 가드레일 — 작성자(Dev) 경계

구현할 때 지키는 경계다. 코드 스타일 상세는 [코드컨벤션](../architecture/DoctorPet-코드컨벤션.md), 설계 근거는 [SA](../architecture/DoctorPet-SA.md)가 정본이다.

## 구현 기준

- Generic Manager·공통 유틸 계층보다 명시적인 도메인 Service 메서드를 우선한다.
- 다른 도메인의 Repository를 직접 호출하지 않는다 — Service를 경유하고, 여러 도메인 조합은 `XxxApplicationService`로 분리한다.
- 모듈 경계를 넘는 참조는 `@ManyToOne`이 아니라 FK 값(Long)으로 한다.
- 트랜잭션은 Service 계층에서, 필요한 범위에서만. 외부 호출(PortOne·LLM·공공데이터)은 트랜잭션 안에 두지 않는다.
- Redis 검색 캐시는 파생 데이터다 — 원천은 항상 MySQL이며, 캐시 미스·장애가 기능 실패로 이어지지 않게 한다.
- Entity 상태 변경은 도메인 메서드로 한다. 상태 머신(SA §5)에 없는 전이를 만들지 않는다.
- Mock/스텁 검증과 실제 인프라(MySQL·Redis·PortOne·LLM) 검증을 로그·기록에서 구분한다.

## 범위 제한

- 하나의 작업(Issue)에서는 하나의 목표만 구현한다.
- 같은 PR에서 문서 대정리, 기능 구현, 인프라 구성, 성능 테스트를 한꺼번에 처리하지 않는다.
- 공통 예외 포맷, DB 스키마, `ApiResponse` 구조, 공통 설정처럼 여러 도메인에 영향을 주는 변경은 별도 chore/architecture Issue로 분리한다.
- k6 부하 테스트·배포 구성 같은 도전 과제는 MVP 기능 구현과 분리한다.

## 범위 확장이 필요해졌을 때 (출구 절차)

범위 밖 구현은 금지지만, 확장이 정말 필요하다고 판단되면 우회하지 말고 아래 절차를 따른다.

1. 구현을 멈추고 **질문 Issue를 만든다** — 확장 근거, 영향받는 문서(PRD·SA 섹션), 예상 변경 범위를 명시한다.
2. 팀 합의를 받는다 (PR 리뷰가 아니라 Issue에서 먼저).
3. 합의되면 **PRD/SA를 먼저 갱신**하고, 그 다음 구현 Issue를 새로 만든다.
4. 합의 전까지 해당 기능의 코드를 작성하지 않는다.

## 확정 결정 준수

AGENTS.md의 "확정된 핵심 결정"을 재논의하거나 우회 구현하지 않는다. 예: 낙관적 락 대신 Redis 분산 락 도입, Caffeine 로컬 캐시 추가, 예약 상태에 결제 상태 통합.
